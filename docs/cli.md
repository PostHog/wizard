The wizard's public command surface follows one rule: if the wizard can pick the variant for you (auto-detection), the command is flat. If you must pick, the command is a family. See [CONTRIBUTING.md](../CONTRIBUTING.md) for the full convention.

## Skill-backed commands

These come from context-mill. Adding a new one is a context-mill PR with a `cli:` block on the skill's `config.yaml` — no wizard release required.

### Flat commands

### `wizard migrate`

Migrate an existing analytics or feature-flag vendor to PostHog. Replaces SDK call sites in-place, removes the source package, and writes a migration report. Replacement-only, doesn't adds new instrumentation.

*Backed by context-mill skill: `migrate-statsig`*

### `wizard revenue-analytics`

Set up Stripe revenue analytics with PostHog

*Backed by context-mill skill: `revenue-analytics-setup`*

### `wizard audit` family

`wizard audit` with no subcommand opens an interactive picker over the children; the default leaf (`all`) is pre-highlighted, so pressing Enter runs it. `wizard audit --help` lists the children as text.

### `wizard audit all` _(default for this family)_

Audit an existing PostHog integration for correctness and best practices

*Backed by context-mill skill: `audit`*

### `wizard audit autocapture`

Audit a PostHog autocapture setup for correctness and cost-optimization opportunities

*Backed by context-mill skill: `audit-autocapture`*

### `wizard audit events`

Audit a PostHog integration's event capture quality and cost-optimization opportunities

*Backed by context-mill skill: `audit-events`*

### `wizard audit feature-flags`

Audit a PostHog integration's feature flag usage for correctness and cost-optimization opportunities

*Backed by context-mill skill: `audit-feature-flags`*

### `wizard audit identify`

## Wizard-native commands

These live in the wizard repo itself, not in context-mill. They're hand-listed in `bin.ts` because they do more than dispatch a single skill (custom CI paths, sub-trees, post-install work, etc.).

### `wizard`

The default integration. Detects the framework, authenticates, runs the agent, and walks the user through their first dashboard.

### `wizard mcp`

Manage PostHog MCP servers across MCP-compatible clients.

### `wizard mcp add`

Install the PostHog MCP server in the user-selected MCP client(s).

### `wizard mcp remove`

Remove the PostHog MCP server from MCP-compatible clients.

### `wizard mcp tutorial`

Walk the user through customizing the role-tailored prompts the MCP server installs.

### `wizard doctor`

Diagnose the PostHog setup for configuration issues, version mismatches, and missing prerequisites.

### `wizard audit web-analytics`

Audit and fix the PostHog web analytics setup. Wizard-native because it does more than run a single skill.

### `wizard source-maps`

Upload source maps for the PostHog error tracking integration. Framework-detected — runs the right uploader for the project.

### `wizard skill list`

List skills in the wizard catalog. Read-only — prints to stdout and exits.

### `wizard skill search <query>`

Search the catalog by name, description, or command path. Read-only.

### `wizard skill <id>`

Run any catalog skill by id, including ones not promoted as top-level commands.

## Internal flags

A handful of flags are accepted on every command but hidden from `--help`. They're for development, debugging, and test infrastructure. See [CONTRIBUTING.md](../CONTRIBUTING.md#internal-flags) for the catalog.
