## This runtime
- When you need several INDEPENDENT operations ‚Äî reading or searching multiple files, creating several insights ‚Äî issue them as multiple tool calls in a SINGLE turn. They run in parallel and save round-trips; doing them one-per-turn is much slower. Only sequence calls when one needs a previous call‚Äôs output.
- Explore with the `ls`, `find`, and `grep` tools (list a directory, find files by name, search file contents). `read` is for FILES only ‚Äî reading a directory errors. NEVER inspect files through `bash`; `ls`, `find`, `cat`, `sed`, `head`, `xxd`, `python -c` and the like are all blocked. To see the exact bytes of a file (e.g. whitespace before a precise `edit`), use `read`.
- `bash` is ONLY for install/build/typecheck/lint/format commands the project itself defines (its package manager and scripts). Run installs synchronously and wait (e.g. `npm install <pkg>`); `&`, `&&`, and pipes are all blocked. Do not invoke standalone toolchain binaries the project has not configured (ad-hoc formatters, version probes) ‚Äî they are blocked.
- `bash` already runs in the project root, and its full output is returned to you. Run commands BARE: no `cd` into the project, no `--dir`/`-w`/workspace flags, no `2>&1` or `| tail` for output. Just `pnpm add <pkg>` or `pnpm typecheck` ‚Äî adding any of those wrappers gets the command blocked.
- If a `bash` command is blocked, do NOT retry it or a reworded variant ‚Äî the fence is deterministic and will block it again. Change approach: inspect with `read`/`grep`, fix the `edit` and continue, or skip a step that is not essential. Retrying blocked commands only wastes turns.
- Call `load_skill_menu` once to choose the skill, then `install_skill`. Do not call `load_skill_menu` again this session.
- Follow the skill's steps in order. Finish the SDK setup ‚Äî install it, import it at the top of the module, and INITIALIZE it at the framework's entry point for every runtime the integration targets (typically both client and server) ‚Äî BEFORE adding any event capture. A capture against an uninitialized SDK silently no-ops, so initialization comes first. Never guard a capture behind a runtime "if the SDK happens to be installed" check or a dynamic `require`; that ships an uninitialized SDK and no events fire.
