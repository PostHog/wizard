## The CLI convention

The wizard's public command surface follows one rule:

> If the wizard can pick the variant for the user (auto-detection), the command
> is flat. If the user must pick, the command is a family.

Examples:

| User has to pick | Wizard picks for them |
|---|---|
| `wizard audit events` | `wizard` (default integration — framework auto-detected) |
| `wizard audit flags` | `wizard revenue` (Stripe is the only provider today) |
| `wizard migrate statsig` | `wizard source-maps` (framework auto-detected) |

When a flat command grows a second variant the user has to pick from, it
becomes a family. `wizard revenue` will become `wizard revenue stripe` /
`wizard revenue chargebee` once a second provider lands.

The full public surface is documented in [`docs/cli.md`](docs/cli.md). That
file is auto-generated by `scripts/generate-cli-docs.cjs` from the
context-mill manifest plus the wizard-native commands — don't hand-edit it.

### When to regenerate `docs/cli.md`

`docs/cli.md` is committed (so people browsing the repo see it) but
generated (so it can't drift from the manifest). Regenerate it whenever
the CLI surface changes:

```bash
pnpm docs:cli
```

Run this after any of the following:

- Updating `cli-manifest.bootstrap.json` (manually or by syncing from
  context-mill).
- Adding, renaming, or removing a wizard-native command in `bin.ts`
  (the file's `NATIVE_COMMANDS` array lives next to the script).
- Pulling a new context-mill release that changes any `cli:` block
  values you care to reflect in our docs.

If you forget, your PR will diff against an out-of-date `docs/cli.md`
the next time someone else does change the surface — better to catch
it locally now.

## Three flag categories

| Category | Visibility | Where declared |
|---|---|---|
| **Global config** | Top-level `--help` | `src/wizard.ts::GLOBAL_OPTIONS` |
| **Per-command config** | Subcommand `--help` | The command's own `options:` block |
| **Internal modes** | Hidden from `--help`, documented below only | `hidden: true` on either of the above |

### When to add a flag

Args are added only when:
1. They're required configuration the wizard can't infer.
2. They're an internal mode flag (always `hidden: true`).
3. They're a filter on a discovery command (`wizard skill list --surface=catalog`)
   where the alternative is a wall of output.

Args are never added because "it might be nice." The CLI surface is curated.

## Internal flags

The wizard ships a set of "internal mode" flags — accepted everywhere, hidden
from `--help`. Each is for development, debugging, or test infrastructure.

| Flag | What it does |
|---|---|
| `--playground` | Launches the TUI primitives playground (an Ink demo harness for layout, picker, content blocks, MCP install screen, etc.). Lives in `src/ui/tui/playground/`. Add new primitive demos here. |
| `--benchmark` | Runs the agent in benchmark mode with per-phase token tracking and detailed timing. Wires up middleware that publishes timing to a JSON report. Use when measuring run-time cost regressions. |
| `--yara-report` | Prints a summary of YARA scanner matches after the agent run finishes (otherwise scanner findings are only logged on block/revert). Useful when iterating on warlock rules. |
| `--local-mcp` | Points the wizard at a local MCP server (`http://localhost:8787/mcp`) instead of `mcp.posthog.com`. Pairs with wizard-workbench's local stack so changes to the MCP surface are testable without deploying. |
| `--ci` | Runs the wizard non-interactively (no TUI). Only available in dev/test builds — disabled in published releases. Used by E2E tests and CI integrations. |
| `--skill <id>` | Dev escape hatch: runs an arbitrary context-mill skill by ID, bypassing the curated public command surface. Combine with `--ci` for headless skill runs. Use when iterating on a skill that isn't promoted to `surface: public` yet. |

Each is also reachable via env var with the `POSTHOG_WIZARD_` prefix
(e.g. `POSTHOG_WIZARD_BENCHMARK=1`).

### Where they're declared

Global internal flags (`--local-mcp`, `--benchmark`, `--yara-report`, `--ci`)
live in `src/wizard.ts` under `GLOBAL_OPTIONS` with `hidden: true`. They apply
to every command automatically — no need to repeat them in per-command option
blocks.

Default-command-only flags (`--playground`, `--skill`) live in
`src/commands/basic-integration/index.ts` on the `basicIntegrationCommand`
options block, also `hidden: true`. They only make sense on the bare
`wizard` invocation because they replace the entire run mode.

### Adding a new internal flag

1. Decide scope: does the flag apply to every command, or only the default
   command? Apply-everywhere flags go in `GLOBAL_OPTIONS`; default-only flags
   go in `basicIntegrationCommand.options`.
2. Set `hidden: true`. Internal flags are never exposed in `--help`.
3. Add a row to the table above explaining what the flag is for.

## Contributor decision tree

When you're adding something new, walk this tree before opening a file.

### "I want to add a new public command"

A public command is one you can promote in our docs and reasonably expect
end users to discover. Three flavors:

- **It's backed by a context-mill skill.** Open a PR to **context-mill**, not
  this repo. Add a `cli:` block to the skill's `config.yaml` with
  `surface: public`. See [context-mill's CONTRIBUTING](https://github.com/PostHog/context-mill/blob/main/CONTRIBUTING.md).
  On the next wizard release after context-mill cuts a release with your
  change, your command shows up — no wizard PR needed.

- **It's wizard-native** (something the wizard runs itself, no agent skill).
  Add a `ProgramConfig` in `src/lib/programs/<name>/`, register it in
  `bin.ts` with `nativeCommandFactory(yourConfig)`. The factory takes care
  of yargs registration, the `--ci` path, options merging.

- **It belongs in a family.** Set `parentCommand` on the `cli:` block (if
  skill-backed) or on the `ProgramConfig` (if wizard-native). The wizard
  groups children under the parent automatically.

### "I want to add a new internal-only command"

You probably don't. Internal behavior is exposed through hidden **flags**,
not hidden subcommands. See "Adding a new internal flag" above.

### "I want to add a new flag"

Walk the three flag categories above. If it's not required configuration,
not an internal mode, and not a discovery-filter — don't add it. Args don't
get added because "it might be nice."

### "I want to add a new subcommand under an existing family"

Same answer as "add a new public command". `wizard audit something-new`
means a new context-mill skill with `cli: { surface: public, parentCommand:
audit, command: something-new }`.

### "I want to change the run mechanics (steps, hooks, content blocks) of an existing skill-backed command"

Edit the `ProgramConfig` in `src/lib/programs/<name>/`. The manifest entry
owns the CLI surface (name, description); the `ProgramConfig` owns
execution. Stay on your side of that fence and you won't break anything.

### "I want to retire an existing command"

Two-repo work:
1. context-mill: change the skill's `surface: public` to `surface: catalog`
   (still reachable via `wizard skill <id>`), or delete the skill entirely
   if it's not worth preserving.
2. wizard: nothing to do — the next release picks up the change from the
   manifest.

For wizard-native commands, remove the `use(...)` line in `bin.ts` and the
file under `src/commands/`. Coordinate with whoever owns docs.
